Continuing the build log — [Field Note #2](/docs/tutorials/kubeops-readonly) proved the RBAC wall was real. This one's about a second wall that shows up only in Claude Code, and why I still don't trust it alone.

<!-- truncate -->

Shipping the Claude Code and Codex adapters this week meant answering a question I'd put off: does either runtime have a config surface that can say "no" to `kubectl delete` before the request leaves the agent? Codex, no — `execpolicy` rules match a literal command prefix, and three trivial forms slip past it: put `--context` first, use an absolute binary path, or wrap the call in `sh -c`. All three return no decision at all, silently.

Claude Code surprised me. `.claude/settings.json` supports `permissions.deny` at the subcommand level — name `kubectl delete`, `kubectl apply`, `helm upgrade` specifically, and Claude Code refuses the tool call outright. Most people evaluating it for ops work don't know this exists; I didn't, until I went looking while building this adapter. The AOH-generated workspace now sets it for every mutation verb in the shared kubectl/helm list.

A literal deny list has the same blind spot Codex's rules do, so I paired it with a `PreToolUse` hook — `kubectl-guard.sh` — that actually parses the command instead of pattern-matching it: strips `sudo`/`env` wrappers, unwraps `sh -c`, strips absolute paths, tokenizes past flags for the real verb. Fail-closed everywhere: bad JSON, missing `jq`, any shell metacharacter, all block. It catches all three forms that beat Codex's rules. It also has a sharp edge I didn't expect until testing it: it blocks *every* compound Bash command, kubectl or not, because it can't safely tokenize syntax it doesn't parse. `kubectl get pods | grep Pending` gets blocked right alongside `kubectl delete pod x`. Documented, not silently patched around.

Here's the honest part. Neither the deny list nor the hook is a boundary — they're guardrails, best-effort, scoped to commands routed through Claude Code's own Bash tool, blind to anything outside that session. The actual boundary is what it was in the Hermes tutorial: a scoped ServiceAccount whose RBAC the API server checks on every request, regardless of what Claude Code's config catches. If the hook has a bug, that's a worse experience, not a security failure — the cluster still says no.

Try it: [the new Claude Code tutorial](/docs/tutorials/kubeops-claude-code) walks through generating the workspace, reading the deny list and hook yourself, and watching both walls hold — one best-effort, one absolute.
