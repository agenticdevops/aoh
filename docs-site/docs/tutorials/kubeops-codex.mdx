# A Read-Only Kubernetes Agent with Codex

[A Read-Only Kubernetes Agent with Claude Code](./kubeops-claude-code) installed the
same pack and binding for Claude Code, which turned out to be the one runtime of the
three that can express subcommand-level deny plus a fail-closed hook. This tutorial
installs the identical pack and binding for **Codex** instead. You'll build a Codex
workspace running the `kubeops-copilot` role, read-only, against
`kind-sresquad-demo` — and see the honest downgrade: Codex has no hook that can block
a tool call, so its guardrail is a best-effort `execpolicy` rules file with three
documented, provable gaps. RBAC carries all the real weight here.

Everything here is copied verbatim from a real run — see
[`docs/demos/adapter-validation-2026-07-16.md`](https://github.com/agenticdevops/aoh/blob/main/docs/demos/adapter-validation-2026-07-16.md)
for the full transcript against the live `kind-sresquad-demo` cluster.

## 1. Generate the workspace

```bash
uv run aoh install --runtime codex collections/core/kubeops \
  --output ./kubeops-codex \
  --binding examples/sresquad-site/bindings/kubeops-sresquad.yaml
```

```text
installed codex workspace in ./kubeops-codex
warning: Codex has no complete Claude-style kubectl guardrail — execpolicy rules are
best-effort prefix matches with known bypass gaps (--context-first, absolute path,
shell wrappers); network access enabled for kubectl; RBAC is the enforcement boundary.
```

Same binding as the Claude Code tutorial — `kubeops-copilot` role,
`kind-sresquad-demo` context, `default` namespace, `access: scoped` (the default).
`aoh install` is the runtime-neutral entrypoint: swap `--runtime codex` for
`--runtime claude-code` or `--runtime hermes` and everything else about the command
stays the same. Note that diagnostic — the adapter prints it on *every* install, not
just when something goes wrong. It's the adapter telling you up front what it cannot
enforce, before you've run a single command.

## 2. Tour the generated artifacts

```text
kubeops-codex/
  .agents/skills/ops-{k8s-service-health-report,node-notready-triage,pending-pod-triage,pod-crashloop-triage}/SKILL.md (+scripts)
  .codex/rules/kubectl-readonly.rules
  .codex/config.toml
  AGENTS.md
  provision.sh
  launch.sh
```

Nothing here touches `~/.codex` — the whole workspace is self-contained under
`--output`.

### `.agents/skills/ops-<skill>/` — frontmatter renamed, not just the directory

Each skill directory is copied under a wrapper name, `ops-<skill>`, and the adapter
rewrites the `name:` line inside the copied `SKILL.md`'s YAML frontmatter to match —
`_rewrite_skill_frontmatter_name` in `src/aoh/adapters/codex.py` parses the
frontmatter, replaces only the `name:` line, and leaves `description:` and the body
untouched. This matters because a directory rename alone does **not** change a
skill's invocation name — Codex reads the name from frontmatter, not the path. Each
skill is invoked as `$ops-<skill>`, e.g. `$ops-pod-crashloop-triage`.

### `AGENTS.md` — the read-only contract and the launch.sh caveat

The generated `AGENTS.md` states the read-only contract plainly:

> You operate under a scoped, read-only Kubernetes identity. Prefer read-only
> inspection; report denials as the system working as intended, not as errors to
> work around. This holds when you start the agent via `./launch.sh` (which exports
> the workspace `KUBECONFIG`); launching codex directly in this directory uses your
> own kubeconfig instead.

That last sentence is the caveat you must not skip — more on it in step 6. It also
states the rules-file honesty up front:

> Be honest about what this is: cluster RBAC is the enforcement boundary; the rules
> file is best-effort. The rules match on a literal command prefix and are known to
> miss `--context`-first invocations, absolute binary paths, and shell-wrapped
> (`sh -c`) commands — RBAC is what actually stops a mutation from succeeding.

### `.codex/config.toml` — workspace-write, and why network access stays on

```toml
model = "gpt-5.4"
model_reasoning_effort = "medium"
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[sandbox_workspace_write]
network_access = true
```

`sandbox_mode = "workspace-write"` lets Codex write inside the workspace directory
without prompting for every file operation — necessary for the pack's skill scripts
to run at all. `network_access = true` is a deliberate trade-off, not an oversight:
`kubectl` needs to reach the cluster API server over the network, and Codex's
sandbox has no finer-grained "allow this host only" knob to carve out just that one
destination. Turning `network_access` off would also break `kubectl`. `approval_policy
= "on-request"` adds a second, coarser layer — Codex will ask before some actions —
but it is not the enforcement boundary either.

Also note what's absent: no `env` block. Codex has no per-project environment
mechanism equivalent to Claude Code's `settings.json` `env.KUBECONFIG`. This is why
`KUBECONFIG` is exported by `launch.sh` at process-start time instead of being baked
into workspace config — see step 6.

Project-local `.codex/config.toml`, hooks, and rules all require Codex to trust this
project before they take effect. Codex will prompt to trust the workspace directory
the first time you launch it here; the generated files assume that trust will be
granted.

### `.codex/rules/kubectl-readonly.rules` — best-effort, with gaps in its own header

```text
# Generated by AOH. Best-effort execpolicy guardrail for kubectl/helm.
#
# THIS IS NOT A SECURITY BOUNDARY. Cluster RBAC is the enforcement
# boundary; this rules file is a best-effort convenience layer on top
# of it. `prefix_rule` matches a literal leading token sequence only,
# so it does NOT catch every way a mutation command can be spelled.
# Verified bypass gaps against codex-cli 0.144.5 `codex execpolicy
# check` (all three fall through to no matched rule):
#   - --context-first invocations, e.g.
#       kubectl --context prod delete pod x
#   - absolute binary paths, e.g.
#       /usr/bin/kubectl delete pod x
#   - shell-wrapped invocations, e.g.
#       sh -c "kubectl delete pod x"
# See AGENTS.md and the AOH design doc for the full threat model.
```

followed by `prefix_rule(pattern=["kubectl", "delete"], decision="forbidden")` for
every kubectl/helm mutation verb and `decision="allow"` entries for the read verbs.
The adapter doesn't hide the gaps in a design doc somewhere else — it writes them
into the file itself, as the first thing anyone reading the rules will see. This is
a best-effort convenience layer, full stop: it matches a literal leading token
prefix only, with no normalization step. Prove it yourself in step 5.

## 3. Provision the identity, then prove RBAC

```bash
./kubeops-codex/provision.sh
```

```text
serviceaccount/aoh-kubeops-sresquad configured
clusterrole.rbac.authorization.k8s.io/aoh-readonly configured
clusterrolebinding.rbac.authorization.k8s.io/aoh-readonly-aoh-kubeops-sresquad unchanged
Scoped read-only kubeconfig written to ./kubeops-codex/kubeconfig
Verify the guardrail: kubectl --kubeconfig ./kubeops-codex/kubeconfig delete pod x  # expect Forbidden
```

Same script shape as the Claude Code tutorial, because both adapters render it
through the shared `src/aoh/adapters/_k8s.py` helpers — the RBAC surface an agent
gets does not depend on which runtime materialized the workspace. If you already ran
`provision.sh` from the Claude Code tutorial against the same binding, this run just
updates the same shared cluster identity (`configured`, not `created`).

### The `can-i` matrix — real output

```text
$ kubectl --kubeconfig ./kubeops-codex/kubeconfig auth can-i get pods
yes

$ kubectl --kubeconfig ./kubeops-codex/kubeconfig auth can-i delete pods
no

$ kubectl --kubeconfig ./kubeops-codex/kubeconfig auth can-i get secrets
no
```

`get secrets` → `no` is the important one: the `ClusterRole` is an explicit resource
allowlist, and `secrets` was deliberately left off it (see the
[Runtime Adapters threat table](../reference/adapters#threat-model-hard-boundary-vs-best-effort-guardrail)
for the full allow/exclude list). This holds identically whether the workspace was
generated for Hermes, Claude Code, or Codex — it's a cluster-side guarantee, not a
runtime one.

And the real mutation attempt, not just `can-i`'s opinion:

```text
$ kubectl --kubeconfig ./kubeops-codex/kubeconfig delete pod -n kube-system coredns-668d6bf9bc-5qhgx
Error from server (Forbidden): pods "coredns-668d6bf9bc-5qhgx" is forbidden: User
"system:serviceaccount:default:aoh-kubeops-sresquad" cannot delete resource "pods" in
API group "" in the namespace "kube-system"
```

## 4. `codex execpolicy check` — the real proof, gaps included

`codex execpolicy check` always exits `0` — the decision lives in the parsed
`"decision"` JSON field, never the process return code. Run against the generated
`.codex/rules/kubectl-readonly.rules` (codex-cli 0.144.5):

```text
$ codex execpolicy check --rules .codex/rules/kubectl-readonly.rules -- kubectl delete pod x
{"matchedRules":[{"prefixRuleMatch":{"matchedPrefix":["kubectl","delete"],"decision":"forbidden"}}],"decision":"forbidden"}

$ codex execpolicy check --rules .codex/rules/kubectl-readonly.rules -- kubectl get pods
{"matchedRules":[{"prefixRuleMatch":{"matchedPrefix":["kubectl","get"],"decision":"allow"}}],"decision":"allow"}
```

That's the rules file working as designed. Now the three documented no-match gap
forms, copied verbatim from the rules file's own header comment:

```text
$ codex execpolicy check --rules .codex/rules/kubectl-readonly.rules -- kubectl --context prod delete pod x
{"matchedRules":[]}

$ codex execpolicy check --rules .codex/rules/kubectl-readonly.rules -- /usr/bin/kubectl delete pod x
{"matchedRules":[]}

$ codex execpolicy check --rules .codex/rules/kubectl-readonly.rules -- sh -c "kubectl delete pod x"
{"matchedRules":[]}
```

`{"matchedRules":[]}` has no `"decision"` key at all — a caller that naively checks
`decision == "forbidden"` and treats anything else as safe would let all three
through. This is the honest framing, stated the same way in the rules file, in
`AGENTS.md`, and in the adapter's diagnostics: the rules file is a **convenience
layer** that catches the common, unadorned case and nothing more; **RBAC is the
boundary**. Unlike Claude Code, Codex has no hook that can actually block a tool call
— Codex's lifecycle hooks are notifications only (`continue: false` is not
supported), so there is no equivalent normalizing blocker available to close these
gaps. If any of these three forms reaches the cluster, it's RBAC — not the rules
file — that stops it, exactly as demonstrated in step 3.

## 5. Launch it

```bash
./kubeops-codex/launch.sh
```

`launch.sh` exports `KUBECONFIG` to the scoped kubeconfig and execs `codex` from
inside the workspace — so `.codex/config.toml`, the rules file, and the skills all
apply automatically.

:::danger[Launching codex directly instead of via launch.sh uses YOUR kubeconfig]
`.codex/config.toml` sets no `env` block — Codex has no per-project environment
mechanism equivalent to Claude Code's `settings.json`. The scoped kubeconfig identity
only applies when the agent is started via `./launch.sh`, which exports the
workspace `KUBECONFIG` before execing `codex`. Running `codex` directly in this
directory (bypassing `launch.sh`) falls through to the host's own `~/.kube/config`
— full host credentials, with only the best-effort execpolicy rules (not RBAC) as a
guardrail. `AGENTS.md` states this caveat inline, but the workspace itself does
nothing to stop you from launching `codex` the wrong way.
:::

A live skill-discovery probe against the workspace confirms the four skills are
exactly what the pack ships, correctly renamed:

```text
$ cd kubeops-codex
$ codex exec --skip-git-repo-check "List the skills available to you by name"
```

```text
Available skills:

- ops-k8s-service-health-report
- ops-node-notready-triage
- ops-pending-pod-triage
- ops-pod-crashloop-triage
```

(`--skip-git-repo-check` was needed because the workspace directory itself isn't a
git repo — Codex refuses to run outside one by default. The real transcript also
listed other skills from the operator's global `$CODEX_HOME`, unrelated to this
pack, omitted here.)

## Why this shape — the honest close

Codex has no Claude-style parser hook. There is exactly **one** hard boundary here —
cluster RBAC, provisioned by `provision.sh`, identical across all three runtimes —
and one best-effort guardrail with three documented, live-verified gaps
(`--context`-first, absolute binary path, `sh -c` wrapper). The rules file and
`approval_policy` are real and useful for the common, unadorned mutation — they just
aren't a substitute for the boundary, and the generated files (rules-file header,
`AGENTS.md`, adapter diagnostics) say so in three separate places rather than
implying more coverage than exists.

- **RBAC is identical across runtimes.** The same `ClusterRole aoh-readonly`
  allowlist, the same `provision.sh` shape, the same `get secrets → no` result —
  none of that depends on Codex, Claude Code, or Hermes.
- **The rules file is convenience, not containment.** It catches
  `kubectl delete pod x` cleanly. It does not catch the same command with a
  `--context` flag first, an absolute path, or a shell wrapper — and it says so in
  its own header comment.
- **`launch.sh` is not optional.** Skipping it silently hands the agent your own
  kubeconfig instead of the scoped one, with no error and no warning at launch time.
