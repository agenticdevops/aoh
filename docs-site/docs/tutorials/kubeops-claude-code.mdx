# A Read-Only Kubernetes Agent with Claude Code

[A Read-Only Kubernetes Agent](./kubeops-readonly) walked through the Hermes
adapter: one wall, cluster RBAC, and Hermes's total lack of a `kubectl`-aware
guardrail. This tutorial installs the exact same pack and binding for **Claude
Code** instead, which changes the picture in one specific way — Claude Code is the
only one of the three shipped runtimes whose configuration can express
subcommand-level deny. You'll see both walls: the RBAC boundary (same as before)
and a second, best-effort guardrail layered in front of it.

Everything here is copied verbatim from a real run — see
[`docs/demos/adapter-validation-2026-07-16.md`](https://github.com/agenticdevops/aoh/blob/main/docs/demos/adapter-validation-2026-07-16.md)
for the full transcript against the live `kind-sresquad-demo` cluster.

## 1. Generate the workspace

```bash
uv run aoh install --runtime claude-code collections/core/kubeops \
  --output ./kubeops-cc \
  --binding examples/sresquad-site/bindings/kubeops-sresquad.yaml
```

```text
installed claude-code workspace in ./kubeops-cc
```

Same binding as the Hermes tutorial — `kubeops-copilot` role, `kind-sresquad-demo`
context, `default` namespace, `access: scoped` (the default). `aoh install` is the
runtime-neutral entrypoint: swap `--runtime claude-code` for `--runtime hermes` or
`--runtime codex` and everything else about the command stays the same.

## 2. Tour the generated artifacts

```text
kubeops-cc/
  .claude/skills/{k8s-service-health-report,node-notready-triage,pending-pod-triage,pod-crashloop-triage}/SKILL.md (+scripts)
  .claude/commands/ops/{...}.md         → /ops:<skill>
  .claude/agents/kubeops-copilot.md
  .claude/settings.json
  .claude/hooks/kubectl-guard.sh
  CLAUDE.md
  provision.sh
  launch.sh
```

Nothing here touches `~/.claude` — the whole workspace is self-contained under
`--output`.

### `.claude/settings.json` — the permissions.deny excerpt

```json
{
  "env": { "KUBECONFIG": "/abs/path/kubeops-cc/kubeconfig" },
  "permissions": {
    "deny": [
      "Bash(kubectl delete:*)",
      "Bash(kubectl apply:*)",
      "Bash(kubectl edit:*)",
      "Bash(kubectl patch:*)",
      "Bash(kubectl create:*)",
      "Bash(kubectl exec:*)",
      "Bash(kubectl drain:*)",
      "Bash(helm upgrade:*)",
      "Bash(helm install:*)"
    ],
    "allow": [
      "Bash(kubectl get:*)",
      "Bash(kubectl describe:*)",
      "Bash(kubectl logs:*)",
      "Bash(./.claude/skills/*)"
    ],
    "defaultMode": "default"
  },
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash", "hooks": [{ "type": "command", "command": "/abs/path/.claude/hooks/kubectl-guard.sh" }] }
    ]
  }
}
```

(Trimmed for readability — the real file lists every mutation verb from the shared
`_k8s.py` allowlist.) This is the under-known capability: Claude Code's own
`permissions.deny` config can name `kubectl delete` specifically and refuse it
before the tool call even happens, no hook required. Most people evaluating Claude
Code for ops work don't know this surface exists.

### `.claude/hooks/kubectl-guard.sh` — the fail-closed backstop

`permissions.deny` matches literal patterns; it doesn't understand `sudo kubectl
delete`, `/usr/bin/kubectl delete`, or `kubectl --context prod delete`. The hook
closes that gap by actually parsing the command: strip `sudo`/`env`/`time`
wrappers, unwrap `sh -c`/`bash -c`, strip a leading absolute path off the binary,
tokenize past flags to find the verb. Wired as a `PreToolUse` hook on the Bash
tool, it fails closed on anything it can't confidently parse — missing `jq`,
malformed JSON, or any shell metacharacter (`| ; & > < $( ) `` \`) in the command
all exit 2 (block).

### `CLAUDE.md` — states both walls honestly

The generated `CLAUDE.md` doesn't let the agent assume the hook is the real
protection. It says, in effect: cluster RBAC is the hard enforcement boundary that
holds regardless of what this workspace's configuration does; the deny list plus
hook is a best-effort, convenience-layer backstop scoped to Claude Code's own Bash
tool, not containment.

## 3. Provision the identity

```bash
./kubeops-cc/provision.sh
```

```text
serviceaccount/aoh-kubeops-sresquad configured
clusterrole.rbac.authorization.k8s.io/aoh-readonly configured
clusterrolebinding.rbac.authorization.k8s.io/aoh-readonly-aoh-kubeops-sresquad unchanged
Scoped read-only kubeconfig written to ./kubeops-cc/kubeconfig
Verify the guardrail: kubectl --kubeconfig ./kubeops-cc/kubeconfig delete pod x  # expect Forbidden
```

Same script shape as the Hermes tutorial, because both adapters render it through
the shared `src/aoh/adapters/_k8s.py` helpers — the RBAC surface an agent gets does
not depend on which runtime materialized the workspace.

## 4. The `can-i` matrix — real output

```text
$ kubectl --kubeconfig ./kubeops-cc/kubeconfig auth can-i get pods
yes

$ kubectl --kubeconfig ./kubeops-cc/kubeconfig auth can-i delete pods
no

$ kubectl --kubeconfig ./kubeops-cc/kubeconfig auth can-i get secrets
no

$ kubectl --kubeconfig ./kubeops-cc/kubeconfig auth can-i create pods
no

$ kubectl --kubeconfig ./kubeops-cc/kubeconfig auth can-i create pods/exec
no
```

`get secrets` → `no` is the important one: the `ClusterRole` is an explicit
resource allowlist, and `secrets` was deliberately left off it (see the
[Runtime Adapters threat table](../reference/adapters#threat-model-hard-boundary-vs-best-effort-guardrail)
for the full allow/exclude list). This holds identically whether the workspace was
generated for Hermes, Claude Code, or Codex — it's a cluster-side guarantee, not a
runtime one.

(`auth can-i get nodes/proxy` reports yes due to a can-i quirk with subresources —
the real request is Forbidden; see the validation evidence doc.)

## 5. The hook, blocking for real

Same adversarial inputs used in the unit tests, run directly against the generated
hook with real stdin and real exit codes:

```text
$ echo '{"tool_input": {"command": "kubectl delete pod x"}}' | bash .claude/hooks/kubectl-guard.sh
kubectl-guard: kubectl delete is a mutation verb; blocked by guardrail
EXIT: 2

$ echo '{"tool_input": {"command": "kubectl --context prod delete pod x"}}' | bash .claude/hooks/kubectl-guard.sh
kubectl-guard: kubectl delete is a mutation verb; blocked by guardrail
EXIT: 2

$ echo '{"tool_input": {"command": "/usr/bin/kubectl delete pod x"}}' | bash .claude/hooks/kubectl-guard.sh
kubectl-guard: kubectl delete is a mutation verb; blocked by guardrail
EXIT: 2

$ echo '{"tool_input": {"command": "sh -c \"kubectl delete pod x\""}}' | bash .claude/hooks/kubectl-guard.sh
kubectl-guard: kubectl delete is a mutation verb; blocked by guardrail
EXIT: 2

$ echo '{"tool_input": {"command": "kubectl get pods -A"}}' | bash .claude/hooks/kubectl-guard.sh
EXIT: 0

$ echo 'not json at all {{{' | bash .claude/hooks/kubectl-guard.sh
kubectl-guard: stdin is not valid JSON; failing closed
EXIT: 2
```

The `--context`-first form, the absolute path, and the `sh -c` wrapper are the same
three forms that provably bypass Codex's execpolicy rules file (`{"matchedRules":[]}`,
no `decision` at all — see the Codex section of
[Runtime Adapters](../reference/adapters)). Claude Code's hook catches all three
because it normalizes the command instead of matching a literal prefix. That's the
under-known part: real command parsing beats prefix matching, and it's sitting in
a shell script most people never look at.

:::danger[Compound commands are blocked outright — read this before you script anything]
The hook's shell-metacharacter check fires on **any** Bash tool call containing
`|`, `;`, `&`, `>`, `<`, `$(...)`, backticks, or backslashes — whether or not
kubectl is even involved. `kubectl get pods | grep Pending` is blocked exactly like
`kubectl delete pod x`. A bare `bash script.sh` (no `-c`) is blocked too, because
the hook only safely unwraps the `sh -c '<command>'` shape. This is deliberate
fail-closed behavior, not a bug — the hook can't safely tokenize syntax it doesn't
parse, so it blocks first. Call the pack's skill scripts directly
(`./script.sh args`, already in `permissions.allow`) or split multi-stage shell
work into separate single-command tool calls instead of chaining them.
:::

## 6. Launch it

```bash
./kubeops-cc/launch.sh
```

`launch.sh` exports `KUBECONFIG` to the scoped kubeconfig and execs `claude` from
inside the workspace — so the settings, hook, and skills all apply automatically.
Ask it to triage the cluster; ask it to delete a pod and watch it explain the
denial as the guardrail working, per `CLAUDE.md`.

## Why this shape

- **Two walls, not one.** RBAC is the hard boundary — it holds no matter what
  Claude Code does. `permissions.deny` + the hook is a second, best-effort wall in
  front of it, and it's a genuinely useful one: it stops the mutation before the
  cluster even sees the request, in the common case.
- **The hook is the strongest guardrail among the three shipped runtimes** — real
  command normalization instead of literal prefix matching — but it is still scoped
  to commands routed through Claude Code's own Bash tool, and it fails closed
  rather than guaranteeing containment of the whole host.
- **RBAC still gets the last word.** If a command somehow reaches the cluster
  despite the hook (a gap the hook doesn't have, but Codex's rules file does — see
  [Runtime Adapters](../reference/adapters)), the API server still rejects it. The
  guardrail reduces noise; the boundary is what makes "physically incapable" true.
