Superpowers for Ops

Gourav Shah  ·  Initcron AI / OpsFlow LLC / School of DevOps

01

Three things break at scale

Not hypothetical — the exact way agentic ops rollouts stall today.

A runbook no one can find, a model no one trusts unsupervised, an agent config copy-pasted per team.

02

The idea: runbook → skill

A skill is instructions plus scripts plus an eval — not just prose.

Instructions, deterministic scripts, and an eval — versioned together, so trust travels with the skill.

03

One pack, many runtimes

Write the skill once. The adapter speaks each runtime's dialect.

Runtime knowledge lives only in the adapter — the pack spec never mentions a runtime by name.

04

If you know Ansible, you know this

Packs are roles. Bindings are inventory. Same split, ops-agent shaped.

One limit to the analogy: AOH organizes, packages, validates, and adapts — it never executes. Runtimes execute.

05

The wedge: cheap models you can trust

Frontier authors once. Cheap model executes many times. Evals are the gate.

Evals are what make handing an incident to a cheap model a defensible decision, not a gamble.

06

A wall between organizing and executing

AOH never touches your cluster, cloud account, or terminal. Runtimes do that.

The wall is deliberate: AOH stays engine-neutral by never crossing into execution itself.

07

Try it

One command validates a pack's referential integrity before anything ships.

Validate runs per pack — it checks the manifest, skills, and every cross-reference resolve.
