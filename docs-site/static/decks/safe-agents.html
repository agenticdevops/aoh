01

Declared intent: kubectl-readonly

A pack declares a RuntimeRequirement — not a prompt instruction, a spec field.

Intent is declared once, in the pack — the adapter is what turns declared intent into an enforced guardrail.

02

Materialized as an RBAC identity

The Hermes adapter provisions a dedicated ServiceAccount and ClusterRole.

A human runs provision.sh once. AOH never touches the cluster itself — it only generates the script.

03

The wall: the API server denies

Same identity, same kubeconfig — one request allowed, one refused.

Not because the agent second-guessed itself — the API server checked the ServiceAccount's RBAC and refused.

04

A separate identity, not a filtered prompt

Distinct from any human operator's credentials — so the audit log tells them apart.

Every action the agent takes is logged under its own identity — distinguishable from anything a human did directly.

05

Honest limit: read-only is not read-nothing

get/list/watch on everything includes every Secret in the cluster.

Production bindings should tighten the role — an aggregated view role plus explicit resources, excluding secrets.

06

Two walls — only one is a boundary

Claude Code / Codex add a runtime guardrail in front of RBAC. Best effort, not enforcement.

The dashed wall may catch it early (nice) or may not (documented gaps) — the solid wall catches it every time, because it's the API server checking an identity, not a runtime reading a string.

07

The demo — live and unscripted

kind-sresquad-demo cluster · scoped kubeconfig · a real kubectl delete.

No agent involved in this proof — just the same guardrail, checked directly against the cluster.
