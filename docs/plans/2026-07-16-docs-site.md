- [ ] **Step 7: GH Actions deploy workflow**

The AOH git repo root IS `/Users/gshah/work/opsflow-sh/experiments/aoh` (nested repo, remote `agenticdevops/aoh`), so the in-repo docs path is `docs-site/` (NOT `experiments/aoh/docs-site/`). `.github/workflows/docs-deploy.yml`:
```yaml
name: Deploy AOH docs to Pages
on:
  push:
    branches: [main]
    paths: ['docs-site/**', '.github/workflows/docs-deploy.yml']
  workflow_dispatch:
permissions:
  contents: read
  pages: write
  id-token: write
concurrency:
  group: pages
  cancel-in-progress: false
jobs:
  build:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: docs-site
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
          cache-dependency-path: docs-site/package-lock.json
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: docs-site/build
  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

`title: Engine-Neutral by Design`. Content (from PROJECT.md decision 2026-07-13 + spec Runtime Boundaries):
- AOH organizes/packages/validates/adapts; runtimes execute. The engine-neutral rule: no runtime-specific concepts in the spec/pack model — runtime knowledge lives only in adapters.

- Declared-intent model: a pack declares a RuntimeRequirement (e.g. `kubectl-readonly`); each adapter maps it to that platform's native guardrail, or documents the gap.
- Mermaid: one Pack fanning out to Hermes / Claude Code / Codex / Goose adapters.

- [ ] **Step 4: `safe-agents.mdx`**

- The safety thesis: declared intent → native guardrail. For a kubernetes target, `kubectl-readonly` materializes as a dedicated RBAC identity (get/list/watch only) + scoped kubeconfig. The agent physically cannot mutate — the cluster enforces, not the prompt.
- Separate identity → audit trail distinguishes agent from human. Honesty note: read-only ≠ read-nothing (can read Secrets on a broad role; production tightens).
- Why not trust the runtime's own guardrails: verified — Hermes's dangerous-command list has no kubectl awareness; `kubectl delete` would run unprompted. So the cluster must be the wall.
- `<Slides src="decks/safe-agents.html" title="Safe Read-Only Agents" />`.
- Mermaid sequence: agent → kubectl get (allowed) ; agent → kubectl delete → API server → Forbidden.
